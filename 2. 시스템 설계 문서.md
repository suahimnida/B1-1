시스템 설계 문서 (문서 1개)
: 타겟 사용자, 업무 문제, 페르소나, 시스템/유저 프롬포트, Few-shot 예시, 검증 전략을 포함하며, 프롬프트 버전(v1 -> v2) 개선 이력과 최종 프롬프트 전문을 담은 문서
  + Zero-shot / Few-shot / 단계적 추론유도 기법의 차이를 구분 및 업무 과업에 적용
  + 페르소나와 시스템 프롬프트가 결과 품질(일관성/안정성/형식 준수)에 미치는 영향
  + 생성 결과와 사실 오류를 테스트로 검증하고 프롬프트를 반복 개선하는 과정을 재현 가능하게 설명
  1) 업무 자동화 과업 1개 선택 및 문제 정의
      -> 회의록 요약(결정사항/Action Items 추출): 어떤 클라우드 솔루션을 사용할지 결정
           -> 타겟 사용자, 입력 데이터 형태, 출력 규격
                -> 입력 템필릿 1개 이상 제시 (학습자가 재사용 가능한 형태로.)
                      Ex) 목적, 대상, 핵심 포인트, 톤, 금지어, 필수 포함 항목 등
  2) 페르소나 정의 및 시스템 프롬프트 설계
       -> 페르소나 구체화 (이름/역할(직무)/전문 분야/말투/금지 사항/우선 순위 (정확성 VS 친절함 등))
            + 역할(페르소나)과 목표(업무 과업) / 출력 형식 규칙 (제목, 불릿, 표현), 안전장치 (정보의 정확성 재확인 혹은 자료 출처)
            + 사실/정책/수치가 포함될 때의 처리 규칙(근거 제시 또는 확인 필요 표기)
  3) Few-shot 예시 최소 3개 포함
       -> 동일 과업에 대해 “좋은 입력 -> 좋은 출력”예시
            + 예시 중 1개는 “모호한 입력” 케이스를 구성하고 AI가 되물어야  하는 지점을 보여주기
  4) 단계적 추론 유도 적용 및 전/후 비교
       -> 같은 과업 입력에 대해 2가지 버전 비교 (v1: 일반 지시 / v2: 단계적 접근을 유도하는 프롬포트(Ex) 요구사항 정리 -> 누락 질문 -> 초안 작성))
            *최종 출력에는 불필요한 장문의 추론 과정이 그대로 노출되지 않도록 “최종 답변은 요약/근거 중심” 규칙을 함께 둘 것
